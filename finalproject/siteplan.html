StrideLab — Individual Project Site Plan

This site plan outlines the design and content strategy for a 3-page, data-driven website project built with HTML, CSS, and JavaScript (no frameworks). This project is separate from the Chamber project.

Site Name

StrideLab

Reason: The project helps users plan walking/running challenges and training weeks. “Stride” matches the theme of steps/distance, and “Lab” implies tracking, experimenting, and improving.

Optional domain idea: stridelab.app (availability not required)

Site Purpose

StrideLab is a practical planning hub for long-distance walking/running challenges. It provides:

  • A curated list of routes/challenges with distance, elevation, difficulty, and required time estimates.
  • Interactive filtering + a detail modal for each route.
  • A simple membership / contact form and a confirmation page that displays submitted data.
  • Weather integration for the selected route location to support training decisions.
  • User preferences saved with localStorage (view mode, favorites, last selected route).
3 pages 15+ items from JSON Modal dialog LocalStorage Fetch + async/await try/catch ES Modules Accessible + responsive

Scenarios

Target audience: people who train for long walks, hikes, or endurance challenges and want simple planning tools.

  1. “I want a 30–45 km route near Sydney. Which one has less elevation so it’s easier on my joints?”
  2. “How long will this route realistically take if I want to finish under 6.5 hours?”
  3. “What’s the weather like for the next 3 days near the route start so I can pick the best day?”

Color Scheme

Two primary colors + usage (applies to the project site later; also used on this plan page):

  • Deep Navy (#0b1220) — site background, header/footer background.
  • Sky Blue (#43b0ff) — headings, links, focus states, buttons.
  • Mint (#2ee59d) — highlights (tags, badges, “favorite” state).
  • Off-White (#e8eefc) — main text.

Typography

  • System UI stack (system-ui, Segoe UI, Roboto, etc.) — fast loading, clean, readable.
  • Usage: headings + body text (consistent). Optional third font not needed to keep performance high.

Planned Pages

  • index.html — Landing page: intro, featured routes, quick filters, CTA, and a small weather panel for the selected route.
  • routes.html — Route library: displays 15+ route cards from JSON, filter/sort, grid/list toggle (saved to localStorage), modal details.
  • join.html — Membership / contact form (GET) + confirmation page (thankyou.html) displaying submitted data.

Note: thankyou.html does not count toward the 3-page requirement.

Wireframe — Mobile (Home)

Header: Logo + Site Name + Hamburger
Hero: Title + short tagline + CTA button
Featured Routes (2–3 cards)
Weather Panel (current + 3-day forecast)
Footer: nav links + attributions + video link

Wireframe — Desktop/Tablet (Home)

Header: Logo + Site Name + Horizontal Nav
Hero: left (text + CTA) / right (image or featured stat)
Main Grid: Featured Routes (left 2/3) + Weather Panel (right 1/3)
Footer: split columns (links / credits / video link)

Data + Dynamic Content Plan (JS requirements mapping)

Data Source

  • Local JSON file: data/routes.json with at least 15 routes.
  • Each route shows at least 4 properties: name, distance_km, elevation_m, difficulty, suburb/start point, estimated_time.

External API

  • OpenWeatherMap API (or similar) to show weather for the selected route start location.

Local Storage

  • Persist user view preference (grid/list) and favorites.
  • Persist last selected route to auto-load its weather on next visit.

Modal Dialog

  • Modal shows route details, tips, and “favorite” action.
  • Accessible: focus trapping, ESC to close, proper labels.
